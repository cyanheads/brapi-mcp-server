# 0.3.0 ÔÇö 2026-04-25

Closes the MVP design surface from `docs/design.md`. The read side (18 tools, 6 services) shipped in 0.1.x / 0.2.x; this release adds the only write tool, the additive resource layer for URI-addressable clients, two workflow prompts that package the canonical multi-step BrAPI flows, and backfills tests for every untested handler.

## Added

- **`brapi_submit_observations` tool** ÔÇö additive write surface for observation rows. Two-phase: `mode: preview` (default) validates rows against the study's `observationvariables` and reports a `routing` breakdown (POST for new, PUT for rows carrying `observationDbId`); `mode: apply` elicits user confirmation via `ctx.elicit`, fans the rows out to POST + PUT in parallel, then re-probes `studies/{id}/observations?pageSize=0` to return the post-write total. Apply requires `ctx.elicit` support OR an explicit `force: true` flag (rejected with `Forbidden` otherwise). Per-row warnings flag unknown variables, empty values, and missing FK fields. Auth scope: `brapi:write:observations`.
- **Six MCP resources** under the `brapi://` scheme ÔÇö additive layer for URI-addressable clients. Each mirrors an existing tool's payload through the default connection:
  - `brapi://server/info` ÔÇö orientation envelope (mirror of `brapi_server_info`).
  - `brapi://calls` ÔÇö raw capability profile from `CapabilityRegistry`.
  - `brapi://study/{studyDbId}` ÔÇö single study with FKs resolved (mirror of `brapi_get_study`).
  - `brapi://germplasm/{germplasmDbId}` ÔÇö single germplasm with attributes + parents (mirror of `brapi_get_germplasm`).
  - `brapi://dataset/{datasetId}` ÔÇö dataset metadata + provenance (rows via `brapi_manage_dataset`).
  - `brapi://filters/{endpoint}` ÔÇö static filter catalog (mirror of `brapi_describe_filters`).
- **Two MCP prompts** ÔÇö multi-step workflow templates that package the canonical BrAPI flows into ready-to-fire user messages:
  - `brapi_eda_study(studyDbId, alias?)` ÔÇö exploratory-data-analysis playbook for one study (orient Ôćĺ variables Ôćĺ coverage Ôćĺ missing Ôćĺ outliers Ôćĺ pedigree Ôćĺ report).
  - `brapi_meta_analysis(germplasmDbIds, traitName, alias?)` ÔÇö cross-study trait meta-analysis (resolve trait Ôćĺ discover studies Ôćĺ harmonize Ôćĺ per-germplasm ├Ś per-study summary Ôćĺ across-study summary Ôćĺ pedigree side-quest Ôćĺ report).
- **`BrapiClient.post()` / `BrapiClient.put()`** ÔÇö generic JSON request helpers shared by the new write tool. Same retry policy, HTTP error reclassification, and SSRF guard as `get()`.

## Changed

- **`brapi_raw_get` cross-origin guard** ÔÇö `rejectCrossOrigin()` now runs against the raw `input.path` before `normalizePath` adds the leading `/`. Previous order let `https://evil/leak` slip past the regex (the leading slash blocked the `/^https?:\/\//` match) and instead concatenate into the registered baseUrl path, where the URL constructor escaped it benignly. Defense-in-depth restored.
- **README** ÔÇö version badge bumped, surface counters updated to `19 Tools ┬Ě 6 Resources ┬Ě 2 Prompts`, new sections for resources and prompts, write-tool entry in the tools table, project structure expanded with `mcp-server/resources/` and `mcp-server/prompts/`.

## Fixed

- **State key validation** ÔÇö switched every service prefix from colon-separated to slash-separated (`brapi:conn:` Ôćĺ `brapi/conn/`, `brapi:capability:` Ôćĺ `brapi/capability/`, `brapi:ds:meta:` / `brapi:ds:rows:` Ôćĺ `brapi/ds/meta/` / `brapi/ds/rows/`, `brapi:ref:ÔÇŽ` Ôćĺ `brapi/ref/ÔÇŽ`). `@cyanheads/mcp-ts-core` 0.7.x storage rejects keys outside `/^[a-zA-Z0-9_.\-/]+$/`, which broke every state-using tool at runtime in HTTP mode (`ServerRegistry.register`, `CapabilityRegistry`, `DatasetStore`, `ReferenceDataCache`). Unit tests passed because `createMockContext` skips key validation; field-testing the HTTP transport surfaced the regression. Also updated the matching state-key assertions in `tests/tools/brapi-connect.tool.test.ts`.
